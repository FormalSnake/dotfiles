# Add App to Formalconf

Guide for integrating a new application into the formalconf theme system.

## Components

1. **Template file** (required) — renders theme colors into the app's config format
2. **Template manifest entry** (required) — registers the template in templates.json
3. **Stow package** (optional) — manages the app's full configuration via GNU Stow

## Step 1: Determine the App's Theme Format

Research the target application's color/theme config format:
- File format: TOML, INI, CSS, Lua, Conf, SCSS, JSON, YAML?
- Color format: hex (#RRGGBB)? Stripped hex? RGB values? RGBA?
- Does it support native light/dark mode switching?

## Step 2: Choose Template Type

- **Single-mode** (`appname.ext.template`): App uses one theme at a time. Most common.
- **Partial-mode** (`appname-dark.ext.template` + `appname-light.ext.template`): App needs separate dark/light files.
- **Dual-mode** (`appname.ext.template` with `"mode": "dual"`): App handles both modes in one file. Rare.

## Step 3: Create the Template File

Create `~/.config/formalconf/templates/appname.ext.template`

Start with a comment header:
```
# {{theme.title}} ({{mode}}) - Generated by FormalConf
```

Map app color settings to formalconf variables with the appropriate modifier:

| App needs | Use modifier | Example output |
|-----------|-------------|----------------|
| `#RRGGBB` | (none) or `.hex` | `#1E1E2E` |
| `RRGGBB` (no hash) | `.strip` | `1E1E2E` |
| `rgb(R,G,B)` | `.rgb` | `rgb(30,30,46)` |
| `rgba(R,G,B,A)` | `.rgba` | `rgba(30,30,46,1)` |
| `R,G,B` | `.decimal` | `30,30,46` |
| Individual 0-255 | `.r`, `.g`, `.b` | `30` |
| Individual 0.0-1.0 | `.red`, `.green`, `.blue` | `0.117647` |

## Step 4: Register in Template Manifest

Add an entry to `~/.config/formalconf/templates/templates.json`:

```json
"appname.ext.template": {
  "version": "1.0.0",
  "installedAt": "<current ISO timestamp>",
  "customOverride": true
}
```

Set `customOverride: true` for user-created templates (prevents CLI from overwriting).

If the rendered file should be copied to additional locations, add `targets`:
```json
"targets": ["~/.config/appname"]
```

For dual-mode templates, add `"mode": "dual"`.

## Step 5: Create Stow Package (Optional)

If managing the app's full configuration (not just theme colors):

1. Create directory: `~/.config/formalconf/configs/appname/`
2. Mirror the home directory structure inside it:
   ```
   configs/appname/.config/appname/config.toml
   configs/appname/.config/appname/keybindings.toml
   ```
3. In the app's main config, source the theme from `~/.config/formalconf/current/theme/appname.ext`

## Reference Templates

Use these existing templates as patterns:

| Pattern | Example | Good for |
|---------|---------|----------|
| Simple hex terminal | `alacritty.toml.template` | TOML/INI with `#hex` colors |
| Stripped hex | `ghostty-dark.theme.template` | Apps wanting hex without `#` |
| CSS theming | `walker.css.template` | GTK/CSS apps |
| SCSS variables | `eww.scss.template` | SCSS-based styling |
| Hyprland-style | `hyprland.conf.template` | Apps using `rgba()` format |
| Dual-mode | `neovim.lua.template` | Apps handling light+dark in one config |
| Partial-mode | `kitty-dark.conf.template` | Separate dark/light config files |

## Step 6: Remind User to Apply

After creating the template and manifest entry, the user must run the formalconf CLI to regenerate all templates and see the new app's themed config.

## Constraints

- Do NOT run formalconf CLI commands
- Set `customOverride: true` in manifest for user-created templates
- After creating, remind user to re-apply their theme
